## 3. 본 파이프라인 5단계 가동

### 1단계: 수집 + 청킹 (2분)

```
819건 / 121초 (병렬 8)
  documents: 1,638건
  chunks (RAG):    14,144  (cost_estimate + non_attachment만)
  bill_text snippets: 2,344 (TAG 단계 보조용, RAG X)
  에러: 0
```

bill_text는 RAG에 안 넣음. (의안원문 통째 청킹은 노이즈만 키움)

### 2단계: RAG Supabase 업로드

```
bills:     819
documents: 1,638
chunks:    14,144
stats:     10
```

### 3단계: 임베딩 + Supabase 업데이트

```
Azure OpenAI text-embedding-3-small (1536차원)
14,044 chunks → embedding 채움
업로드: 13,844 / 14,044  (HNSW 인덱스 timeout 약 1.5% 손실)
```

### 4단계: TAG 추출 (Gemini)

**`extract_tag_structures.py` 대대적 수정:**
- `--skip-bill-text`: 의안원문 분석 스킵 (불필요 결론)
- `--skip-existing`: 이미 처리된 의안 건너뜀 (재시작 가능)
- `--parallel N`: ThreadPoolExecutor 의안 단위 병렬
- 결과 append 모드 (skip-existing과 호환)
- Gemini list 응답 처리

**의안원문 빼는 결정의 근거:**
- 미첨부사유서 10건 직접 분석한 결과, 한국 국회 표준 양식이
  "조·항(조제목) / 주요내용 / 미첨부 근거 / 상세 사유"를 전부 포함.
- 의안원문 별도 분석 없이 미첨부사유서/추계서 본문만으로 정보 95% 커버.
- 처리 시간 5배 절감.

**400건 누적 결과 (100건 + 300건 + 150건 재시도):**

## 4. 마주친 이슈와 해결

### A. Python stdout 버퍼링
- 백그라운드에서 진행상황이 안 보임
- 해결: `PYTHONUNBUFFERED=1 python3 -u`

### B. Gemini rate limit (429)
- `gemini-3-pro-preview` + 병렬 5 + sleep 0.3 → 후반부 429 폭주
- 해결: `gemini-2.5-pro` 변경 + 병렬 3 + sleep 1.0 → 100% 성공률

### C. amount_thousand integer 범위 초과
- 22억 천원 = 2.2조 원짜리 추계서가 PostgreSQL `integer` 한도(±21억) 초과
- 해결:
  ```sql
  ALTER TABLE public.cost_estimate_amounts
    ALTER COLUMN amount_thousand TYPE bigint;
  ```

### D. assembly_bills 결과 라벨 컬럼 누락
- 미리 설계되지 않았던 `has_cost_estimate / has_non_attachment` 필요성 발견
- 해결: 스키마 + upload 코드에 추가 (ALTER TABLE 실행 완료)

```

🔴 제3조(도지사 책무) - 직접지원 - mandatory
```

**자동 생성 추계서:**

- 계획 운영비 = 연구용역비 + 위원회 운영비 + 위탁 수수료

**필요 변수 (KOSIS 매핑 후보):**
도내 공공임대주택 가구 수, 가구당 월 평균 공용관리비,

**한계:** 연도별 금액은 `null` (구체 통계 없을 땐 정직히 비움)

## 7. 벡터 검색 정확도 개선 (SQL RPC 추가)

### 문제
초기 시뮬레이션 시 RAG 검색이 정렬 없이 LIMIT 5만 걸어서 부정확.

### 해결
Supabase에 `match_assembly_chunks` SQL 함수 추가 (사용자가 SQL Editor에서 실행 완료):

```sql
create or replace function match_assembly_chunks(
  query_embedding vector(1536),
  match_count int default 5,
  filter_source text default null,
  filter_doc_type text default null
)
returns table (
  chunk_id text, bill_id text, bill_no text, bill_name text,
  source text, document_type text, content text, similarity float
) ...
order by embedding <=> query_embedding
limit match_count;
```

이제 PostgREST `POST /rest/v1/rpc/match_assembly_chunks` 호출로
진짜 코사인 유사도 ANN 검색 가능.
