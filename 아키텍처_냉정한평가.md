# 비용추계자동화시스템 — 아키텍처 냉정한 평가

## 결론부터

**방향은 맞다. 하지만 "실서비스"라고 부르기엔 6개 구멍이 있다.**

현재 구조(Supabase Storage + Postgres + pgvector)는 최신 RAG 관행에 부합하고, 초기 프로덕션 구조로 과하지 않다. 이건 확실하다. 그런데 실제로 "인건비", "최저임금" 같은 쿼리를 날리면 **제대로 못 찾을 가능성이 높다.** 이유를 하나씩 짚겠다.

| 항목 | 평가 |
|------|------|
| 원본 파일(Storage)과 벡터(Postgres) 분리 | ✅ Supabase 공식 권장 패턴 그대로 |

## 🔴 구멍 6개 (냉정하게)

### 1. 한국어 Full-Text Search가 작동하지 않는다

> ⚠️ **이것이 가장 큰 문제다.** Supabase Hybrid Search 문서의 `tsvector`는 **영어 기준**이다.

현재 스키마에 `tsvector` 컬럼이 **없다.** 추가하더라도:

```sql
-- Supabase 문서 예시
fts tsvector generated always as (to_tsvector('english', content)) stored
```

이걸 그대로 쓰면 **"인건비", "최저임금", "소비자물가지수"를 못 찾는다.**

PostgreSQL 내장 `to_tsvector()`는 한국어 형태소 분석을 지원하지 않는다. 공백 기준으로만 토큰화하기 때문에 "소비자물가지수"를 "소비자" + "물가" + "지수"로 분리하지 못한다.

**해결 방안 (2가지):**

| 방법 | 장점 | 단점 |
|------|------|------|
| **PGroonga 확장** | CJK 언어 N-gram 지원, Supabase에서 활성화 가능 | 별도 인덱스 관리 |
| **pg_trgm (Trigram)** | 설정 간단, LIKE 검색 대체 | 대량 데이터에서 성능 열화 |

**추천:** `PGroonga`를 우선 확인하고, Supabase 프로젝트에서 활성화 가능한지 체크.
불가능하면 `pg_trgm` + `ILIKE` 조합으로 시작.

### 2. IVFFlat 인덱스는 지금 단계에서 위험하다

현재 스키마:
```sql
create index if not exists assembly_chunks_embedding_idx
on public.assembly_chunks
using ivfflat (embedding vector_cosine_ops)
with (lists = 100);
```

> ⚠️ IVFFlat은 **데이터가 이미 충분히 있어야 제대로 작동한다.** 50건 시드 데이터에 `lists = 100`은 의미가 없다.

Supabase 공식 문서 기준:
- **HNSW**: High accuracy, read-heavy workloads, low-latency → **우리 케이스에 맞음**
- **IVFFlat**: 100k~10M rows, lower memory → 나중에 스케일할 때

**지금은 HNSW로 바꿔야 한다:**

```sql
create index if not exists assembly_chunks_embedding_idx
on public.assembly_chunks
using hnsw (embedding vector_cosine_ops);
```

IVFFlat은 10만건 이상 누적된 후에 고려해도 늦지 않다.

### 3. 테이블/산식 보존 Chunking 전략이 없다

현재 chunking 코드:

```python
def split_chunks(text: str, *, max_chars: int = 1200, overlap: int = 160) -> list[str]:
    normalized = re.sub(r"\s+", " ", text).strip()
```

이 방식의 문제:
- **표(table)를 그냥 잘라버린다.** 비용추계서의 핵심은 산출 내역 표인데, 행 중간에서 잘리면 맥락이 완전히 깨진다.
- **산식이 분리된다.** "대상인원 × 단가 × 12개월 = 연간 비용" 같은 산식이 chunk 경계에 걸리면 의미가 사라진다.
- `max_chars = 1200`은 대략 400~500 토큰 수준으로, 추천 범위(600~1000 tokens)보다 **작다.**

**최신 RAG 모범 사례 기준 해결 방안:**

```
1. 표 → 통째로 하나의 chunk로 보존 (헤더 포함)
2. 산식 → 산식 전후 맥락을 함께 포함
3. 일반 텍스트 → 800 tokens, 200 tokens overlap
4. 조문 단위 → "제N조" 패턴으로 분할 (이미 부분적으로 구현됨)
```

특히 **표는 Markdown으로 변환해서 chunk에 넣는 것**이 2025/2026 RAG의 표준 접근이다.

### 4. Reranker가 설계에 없다

현재 흐름:
```
Query → Embedding → pgvector 유사도 검색 → Top K 반환 → LLM에 전달
```

2025/2026 RAG의 표준 흐름:
```
Query → Embedding → pgvector Top 50~100 → Reranker(Cross-Encoder) → Top 5~10 → LLM
```

> 💡 **Reranker는 RAG 정확도를 올리는 가장 효과적인 단일 방법이다.** 비용추계서는 정확도가 99%여야 하는 도메인인데, 1단계 검색만으로는 부족하다.

추천:
- **Cohere Rerank API** 또는 **Jina Reranker** (API 기반, 인프라 부담 없음)
- 또는 Self-hosted `bge-reranker-v2-m3` (한국어 지원)

당장은 안 만들어도 되지만, **아키텍처에 "reranker 레이어가 들어갈 자리"는 지금 잡아놔야 한다.**

### 5. `assembly_ingestion_jobs` 테이블이 없다

최종 추천 구조에 적어뒀는데, 실제 스키마에는 없다.

이 테이블이 없으면:
- 어떤 의안이 수집 완료되었는지 추적 불가
- 임베딩 생성이 어디까지 진행되었는지 모름
- 재시도/실패 처리 불가

create table if not exists public.assembly_ingestion_jobs (
  id bigserial primary key,
  bill_id text references public.assembly_bills(bill_id) on delete cascade,

  error_message text,
  metadata jsonb,
  started_at timestamptz,
  completed_at timestamptz,
  created_at timestamptz not null default now()

### 6. chunk 메타데이터에 `age`, `committee`, `propose_date` 누락

현재 `assembly_chunks` 스키마:
```
chunk_id, bill_id, bill_no, bill_name, source,
document_name, document_type, chunk_index, content,
hash_vector, embedding
```

빠진 것: `age`, `committee`, `propose_date`, `storage_path`

이 필드들은 검색 시 **필터링에 필수적**이다:
- "22대 국회 비용추계서에서만 검색"
- "보건복지위원회 관련 의안만"
- "2024년 이후 발의 건만"

Supabase Semantic Search 문서:
> "Push the filter into the SQL function so the planner can combine it with the vector predicate."

chunk 테이블에 필터 대상 컬럼이 없으면, JOIN을 해야 하고, 인덱스 활용이 떨어진다.

## 🔍 Hybrid Search 함수 (한국어 대응, PGroonga 기준)

```sql
create or replace function hybrid_search_chunks(
  query_text text,
  query_embedding extensions.vector(1536),
  match_count int,
  filter_document_type text default null,
  filter_age integer default null,
  full_text_weight float default 1.0,
  semantic_weight float default 1.0,
  rrf_k int default 50
)
returns table (
  chunk_id text,
  bill_id text,
  bill_no text,
  bill_name text,
  document_type text,
  chunk_index integer,
  content text,
  combined_score float
)
language sql stable
as $$
with keyword_results as (
  select
    c.chunk_id,
    row_number() over (order by pgroonga_score(c.tableoid, c.ctid) desc) as rank_ix
  from assembly_chunks c
  where c.content &@~ query_text
    and (filter_document_type is null or c.document_type = filter_document_type)
    and (filter_age is null or c.age = filter_age)
  limit least(match_count, 30) * 2
),
semantic_results as (
  select
    c.chunk_id,
    row_number() over (order by c.embedding <=> query_embedding) as rank_ix
  from assembly_chunks c
  where (filter_document_type is null or c.document_type = filter_document_type)
    and (filter_age is null or c.age = filter_age)
  order by c.embedding <=> query_embedding
  limit least(match_count, 30) * 2
)
select
  c.chunk_id,
  c.bill_id,
  c.bill_no,
  c.bill_name,
  c.document_type,
  c.chunk_index,
  c.content,
  (coalesce(1.0 / (rrf_k + kw.rank_ix), 0.0) * full_text_weight +
   coalesce(1.0 / (rrf_k + sm.rank_ix), 0.0) * semantic_weight)::float as combined_score
from keyword_results kw
full outer join semantic_results sm on kw.chunk_id = sm.chunk_id
join assembly_chunks c on coalesce(kw.chunk_id, sm.chunk_id) = c.chunk_id
order by combined_score desc
limit least(match_count, 30);
$$;
```

## 임베딩 모델 선택

| 모델 | 차원 | 한국어 | 비용 | 추천 |
|------|------|--------|------|------|
| OpenAI `text-embedding-3-small` | 1536 | 양호 | $0.02/1M tokens | ✅ **초기 추천** |
| OpenAI `text-embedding-3-large` | 3072 | 좋음 | $0.13/1M tokens | 비용 대비 불필요 |
| `multilingual-e5-large` | 1024 | 우수 | Self-hosted (GPU 필요) | 나중에 fine-tuning 시 |
| `bge-m3` | 1024 | 우수 | Self-hosted | 하이브리드 검색 내장 |

**판단:** `text-embedding-3-small`로 시작. 한국어 성능 "양호" 수준이고, 비용추계서는 법률/행정 한국어라 범용 모델로도 충분하다. 정확도 문제 생기면 fine-tuned 모델로 교체.
